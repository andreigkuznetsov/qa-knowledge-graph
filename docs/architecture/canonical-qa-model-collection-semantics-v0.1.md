# Canonical QA Model Collection Semantics Specification v0.1

## 1. Статус документа

Настоящий документ определяет нормативную семантику коллекций для **Canonical QA Model v0.1**.

Документ дополняет существующую структурную схему Canonical QA Model и устанавливает правила, необходимые для:

* семантического сравнения полных состояний Artifacts;
* проверки Canonical Changes;
* детерминированного построения Proposed Model;
* сравнения Base Model и Proposed Model;
* будущих возможностей анализа изменений;
* одинаковой интерпретации модели разными производителями и потребителями.

Настоящий документ не изменяет:

* структуру Canonical QA Model v0.1;
* допустимые типы Node;
* допустимые типы Relationship;
* Canonical Identity;
* правила Validation Core;
* Canonical Change Semantic Contract;
* существующие JSON Schema ограничения.

Документ определяет только **семантическое значение порядка элементов коллекций**.

---

## 2. Нормативные термины

Термины **ДОЛЖЕН**, **НЕ ДОЛЖЕН**, **СЛЕДУЕТ**, **МОЖЕТ** используются как нормативные требования.

### 2.1. Упорядоченная коллекция

**Упорядоченная коллекция** — коллекция, в которой позиция каждого элемента является частью канонического значения.

Две упорядоченные коллекции семантически равны только тогда, когда:

1. они содержат одинаковое количество элементов;
2. элементы на каждой соответствующей позиции семантически равны.

Изменение порядка элементов упорядоченной коллекции является изменением состояния модели.

### 2.2. Неупорядоченная коллекция

**Неупорядоченная коллекция** — коллекция, в которой позиция элемента не является частью канонического значения.

Две неупорядоченные коллекции семантически равны, когда они содержат один и тот же набор семантически равных элементов независимо от порядка представления.

Изменение только порядка элементов неупорядоченной коллекции не является изменением состояния модели.

### 2.3. Коллекция с явным порядком

**Коллекция с явным порядком** — коллекция, семантический порядок которой определяется специальным свойством элемента, а не физической позицией элемента в JSON-массиве.

Для такой коллекции физический порядок массива не является каноническим, если все элементы имеют корректное и однозначное значение свойства порядка.

### 2.4. Путь коллекции

Путь коллекции — логическое расположение массива или другой коллекции внутри Canonical QA Model.

Путь определяется структурой модели, а не конкретными индексами элементов.

Например:

```text
nodes[*].tags
```

обозначает коллекцию `tags` любого Node.

---

## 3. Основное правило

Для Canonical QA Model v0.1 действует следующее правило:

> Любая коллекция является упорядоченной, если настоящий документ явно не определяет для неё неупорядоченную семантику или семантику явного порядка.

Это правило распространяется на:

* коллекции, явно описанные JSON Schema;
* вложенные коллекции;
* массивы внутри открытых объектов;
* массивы внутри `metadata`;
* массивы внутри `details`;
* массивы внутри `parameters`;
* расширения модели, допустимые текущей схемой.

Производитель или потребитель модели НЕ ДОЛЖЕН самостоятельно выводить семантику коллекции из:

* имени свойства;
* типа Java-коллекции;
* текущего порядка элементов;
* `uniqueItems`;
* поведения пользовательского интерфейса;
* порядка сериализации;
* поведения конкретного JSON-парсера;
* примеров в тестовых данных.

Отсутствие отдельного правила означает **упорядоченную семантику**.

---

## 4. Семантика стандартных коллекций Canonical QA Model v0.1

### 4.1. Корневые коллекции Nodes и Relationships

Коллекции:

```text
nodes
relationships
```

являются **неупорядоченными**.

Позиция Node или Relationship внутри корневого массива не является частью канонического состояния модели.

Элементы идентифицируются по Canonical Identity:

* Node — по Node ID;
* Relationship — по Relationship ID.

Перестановка Nodes или Relationships без изменения их содержимого не изменяет семантическое состояние Canonical QA Model.

При этом конкретный производитель МОЖЕТ использовать стабильный порядок сериализации для детерминированного вывода, диагностик или тестирования. Такой порядок не становится частью семантики модели.

---

### 4.2. Tags

Коллекции:

```text
nodes[*].tags
```

а также другие коллекции, прямо объявленные как `tags`, являются **неупорядоченными**.

Tags интерпретируются как множество уникальных значений.

Следующие представления семантически равны:

```json
["api", "security"]
```

```json
["security", "api"]
```

Регистр и содержимое каждого значения при этом сохраняют значение.

Например:

```json
["API"]
```

и

```json
["api"]
```

не являются семантически равными.

Если схема запрещает дублирование tags, наличие дубликатов является структурной ошибкой модели. Оно не преобразует коллекцию в мультимножество.

---

### 4.3. Source References

Коллекции:

```text
nodes[*].sourceReferences
relationships[*].sourceReferences
```

являются **неупорядоченными**.

Порядок перечисления источников не изменяет смысл Artifact.

Два набора Source References семантически равны, если каждый Source Reference из одной коллекции имеет семантически равный элемент в другой коллекции.

Сравнение Source References выполняется по их полному каноническому содержимому, а не только по отображаемому названию или позиции.

Если Source Reference имеет собственную Canonical Identity, сопоставление СЛЕДУЕТ выполнять по этой идентичности с последующим сравнением полного состояния.

Если собственной Canonical Identity нет, сопоставление выполняется по полному семантическому значению элемента.

---

### 4.4. Scenario: given, when и then

Коллекции:

```text
nodes[*].scenario.given
nodes[*].scenario.when
nodes[*].scenario.then
```

являются **упорядоченными**.

Порядок элементов отражает последовательность условий, действий или ожидаемых результатов сценария.

Следующие представления не являются семантически равными:

```json
{
  "when": [
    {"id": "STEP-1", "text": "Пользователь отправляет запрос"},
    {"id": "STEP-2", "text": "Система выполняет проверку"}
  ]
}
```

```json
{
  "when": [
    {"id": "STEP-2", "text": "Система выполняет проверку"},
    {"id": "STEP-1", "text": "Пользователь отправляет запрос"}
  ]
}
```

Изменение порядка элементов `given`, `when` или `then` является изменением Artifact State.

---

### 4.5. Preconditions

Коллекции:

```text
nodes[*].testImplementation.preconditions
```

являются **упорядоченными**.

Canonical QA Model v0.1 интерпретирует их как последовательность условий подготовки и проверки исходного состояния.

Перестановка preconditions считается изменением состояния Test Artifact.

Это правило не утверждает, что каждая precondition обязательно выполняется как отдельная техническая команда. Оно только фиксирует, что сохранённая автором последовательность является частью канонического значения.

---

### 4.6. Test Steps

Коллекции:

```text
nodes[*].testImplementation.steps
```

являются **коллекциями с явным порядком**.

Семантический порядок Test Steps определяется значением свойства:

```text
order
```

Физическая позиция Test Step в JSON-массиве не является канонической, когда:

* каждый Step содержит корректное значение `order`;
* значения `order` уникальны внутри одного Test;
* порядок может быть однозначно восстановлен.

Следующие массивы семантически эквивалентны при равенстве полного содержимого Steps:

```json
[
  {
    "order": 1,
    "action": "Открыть форму"
  },
  {
    "order": 2,
    "action": "Отправить данные"
  }
]
```

```json
[
  {
    "order": 2,
    "action": "Отправить данные"
  },
  {
    "order": 1,
    "action": "Открыть форму"
  }
]
```

При семантическом сравнении Steps ДОЛЖНЫ быть сопоставлены по `order`.

Изменение значения `order` является изменением состояния Step и может изменить порядок выполнения.

Если:

* значение `order` отсутствует;
* значение `order` некорректно;
* два Steps имеют одинаковое значение `order`;
* порядок невозможно восстановить однозначно,

модель считается не соответствующей требованиям Canonical QA Model v0.1 и ДОЛЖНА быть отклонена Validation Core или соответствующим валидатором.

Потребитель НЕ ДОЛЖЕН использовать физическую позицию элемента для исправления или угадывания некорректного `order`.

---

### 4.7. Scenario Steps и Parameters

Когда Scenario Step представлен отдельным объектом, коллекция таких Steps является **упорядоченной**, если для неё не определено отдельное свойство явного порядка.

Коллекции внутри:

```text
nodes[*].scenario.(given|when|then)[*].parameters
```

подчиняются общему правилу:

* объектные свойства сравниваются без учёта порядка свойств;
* массивы внутри `parameters` являются упорядоченными;
* вложенные массивы также являются упорядоченными;
* перестановка элементов массива внутри `parameters` считается изменением состояния.

---

### 4.8. Metadata

Объекты `metadata` могут содержать открытые и расширяемые значения.

Для Canonical QA Model v0.1 устанавливаются следующие правила:

1. Порядок свойств JSON-объекта не имеет семантического значения.
2. Присутствие свойства имеет семантическое значение.
3. Отсутствующее свойство отличается от свойства со значением `null`.
4. Отсутствующее свойство отличается от пустого объекта.
5. Отсутствующее свойство отличается от пустого массива.
6. Любой массив внутри `metadata` является упорядоченным.
7. Вложенный массив внутри другого массива также является упорядоченным.
8. Порядок элементов массива является частью значения.
9. Потребитель НЕ ДОЛЖЕН автоматически сортировать массивы внутри `metadata`.
10. Потребитель НЕ ДОЛЖЕН интерпретировать массив как множество только на основании одинакового типа его элементов.

Например:

```json
{
  "metadata": {
    "owners": ["qa", "developer"]
  }
}
```

и

```json
{
  "metadata": {
    "owners": ["developer", "qa"]
  }
}
```

не являются семантически равными в Canonical QA Model v0.1.

Для объявления конкретного массива внутри `metadata` неупорядоченным требуется новая нормативная спецификация Canonical QA Model.

---

### 4.9. Details

Для открытых объектов:

```text
nodes[*].technicalImplementation.details
nodes[*].check.details
```

действуют те же правила, что и для `metadata`.

В частности:

* свойства объектов не зависят от порядка представления;
* массивы являются упорядоченными;
* вложенные массивы являются упорядоченными;
* перестановка элементов считается изменением;
* отсутствующее значение, `null`, пустой массив и пустой объект различаются.

---

### 4.10. Parameters

Для объектов и значений:

```text
parameters
```

действуют следующие правила:

* порядок свойств объекта не имеет значения;
* массивы являются упорядоченными;
* вложенные массивы являются упорядоченными;
* порядок элементов массива является частью канонического значения;
* автоматическая сортировка или преобразование массива в множество запрещены.

---

### 4.11. Dependencies и References

Если Canonical QA Model содержит коллекции идентификаторов, описывающие:

* зависимости;
* ссылки;
* связанные Artifacts;
* участников отношения;
* применимые типы,

они являются **неупорядоченными** только тогда, когда конкретное поле явно определено настоящим документом или другой нормативной спецификацией как множество.

В противном случае применяется основное правило: коллекция является упорядоченной.

Наличие `uniqueItems: true` само по себе не объявляет коллекцию неупорядоченной.

## 6. Семантическое сравнение коллекций

### 6.1. Упорядоченные коллекции

Упорядоченные коллекции сравниваются позиционно.

Коллекции равны, если:

* размер одинаков;
* элемент с индексом `0` семантически равен элементу с индексом `0`;
* элемент с индексом `1` семантически равен элементу с индексом `1`;
* то же правило выполняется для всех остальных индексов.

Потребитель НЕ ДОЛЖЕН предварительно сортировать такую коллекцию.

### 6.2. Неупорядоченные коллекции

Неупорядоченные коллекции сравниваются по членству.

Физическое расположение элементов не учитывается.

Алгоритм сравнения ДОЛЖЕН обеспечивать одинаковый результат независимо от:

* порядка входных элементов;
* реализации списка;
* реализации множества;
* порядка обхода map или set;
* производителя JSON.

Настоящий документ не предписывает конкретный алгоритм сопоставления.

### 6.3. Дубликаты

Если коллекция по схеме должна содержать уникальные элементы, наличие дубликатов является ошибкой валидности модели.

Семантический компаратор НЕ ДОЛЖЕН молча удалять дубликаты.

До сравнения состояния модель ДОЛЖНА пройти соответствующую структурную или семантическую валидацию.

### 6.4. Null, absence и empty

Для любой коллекции различаются:

* отсутствующее свойство;
* свойство со значением `null`;
* пустая коллекция;
* непустая коллекция.

Они не являются семантически эквивалентными, если отдельная нормативная спецификация Canonical QA Model явно не определяет обратное.

Например:

```json
{}
```

```json
{
  "tags": null
}
```

```json
{
  "tags": []
}
```

представляют три разных состояния.

JSON Schema default не материализует отсутствующее свойство и не делает эти состояния равными.

---

## 7. Детерминированное представление

Семантическая неупорядоченность коллекции не запрещает детерминированное представление.

Производитель МОЖЕТ выводить неупорядоченные коллекции в стабильном порядке для:

* воспроизводимой сериализации;
* тестирования;
* формирования fingerprint;
* диагностики;
* сравнения файлов;
* удобства ревью.

Однако такой порядок является правилом представления, а не частью семантического состояния.

Изменение только порядка представления неупорядоченной коллекции:

* не создаёт Canonical Change;
* не делает `MODIFIED` значимым;
* не должно приводить к неравенству Artifact States.

---

## 8. Взаимодействие с Canonical Change

Canonical Change implementation ДОЛЖНА использовать настоящую спецификацию при сравнении Artifact States.

Canonical Change implementation НЕ ДОЛЖНА:

* объявлять собственную семантику коллекций;
* считать все JSON-массивы упорядоченными без учёта настоящего документа;
* считать все массивы неупорядоченными;
* выводить семантику из `uniqueItems`;
* нормализовать массивы без нормативного основания;
* менять порядок элементов внутри хранимого Artifact State;
* материализовывать schema defaults;
* стирать различия между absence, `null` и empty.

Если модель заявляет версию `v0.1`, применяются правила настоящего документа.

Если модель заявляет неизвестную или неподдерживаемую версию, Canonical Change implementation ДОЛЖНА классифицировать декларацию в соответствии с Canonical Change Semantic Contract и НЕ ДОЛЖНА угадывать семантику коллекций.

---

## 9. Взаимодействие с Validation Core

Validation Core продолжает владеть:

* структурной валидностью Canonical QA Model;
* семантической валидностью Canonical QA Model;
* уникальностью Canonical Identity;
* валидностью Relationship endpoints;
* ограничениями типов;
* обязательностью и допустимостью свойств.

Настоящий документ не переносит эти обязанности в Canonical Change.

Canonical Change использует коллекционную семантику только для:

* сравнения Artifact States;
* проверки before-state;
* проверки значимости `MODIFIED`;
* детерминированного построения и проверки Proposed Model.

Некорректная коллекция не должна автоматически исправляться при сравнении.

---

## 10. Расширение модели

Новое поле-коллекция, добавленное в совместимую версию Canonical QA Model, по умолчанию считается упорядоченным.

Чтобы объявить новую коллекцию неупорядоченной или имеющей явный порядок, спецификация новой версии ДОЛЖНА определить:

* путь коллекции;
* тип семантики;
* способ идентификации или сопоставления элементов;
* правила дубликатов;
* правила вложенных коллекций;
* поведение при недостаточной информации.

Изменение существующей коллекции:

```text
ordered → unordered
```

или:

```text
unordered → ordered
```

является изменением семантики Canonical QA Model и требует новой версии семантической спецификации.

---

## 11. Совместимость версий

Настоящая спецификация применяется только к:

```text
Canonical QA Model v0.1
```

Она не определяет автоматическую совместимость между версиями.

Canonical Change Set не должен:

* сравнивать Artifact States разных версий;
* переносить значения между версиями;
* применять правила v0.1 к другой версии;
* выполнять миграцию модели.

Base Model, before-state, after-state и Proposed Model должны интерпретироваться в рамках одной поддерживаемой версии Canonical QA Model.

## 13. Решение исходных открытых вопросов

Для Canonical QA Model v0.1 устанавливаются окончательные ответы:

| Вопрос                                                             | Решение           |
| ------------------------------------------------------------------ | ----------------- |
| Являются ли `tags` упорядоченными?                                 | Нет               |
| Являются ли Node и Relationship `sourceReferences` упорядоченными? | Нет               |
| Являются ли `given`, `when`, `then` упорядоченными?                | Да, по позиции    |
| Являются ли `preconditions` упорядоченными?                        | Да, по позиции    |
| Определяется ли порядок Test Steps позицией массива?               | Нет               |
| Определяется ли порядок Test Steps свойством `order`?              | Да                |
| Как сравниваются массивы внутри `metadata`?                        | Как упорядоченные |
| Как сравниваются массивы внутри `details`?                         | Как упорядоченные |
| Как сравниваются массивы внутри `parameters`?                      | Как упорядоченные |
| Как обрабатывается неизвестная коллекция?                          | Как упорядоченная |
| Делает ли `uniqueItems` коллекцию неупорядоченной?                 | Нет               |
| Различаются ли absence, `null` и empty?                            | Да                |

## 14. Решение о готовности

Настоящая спецификация устанавливает достаточную и детерминированную семантику для всех коллекций Canonical QA Model v0.1.

Для каждой коллекции теперь существует одно из следующих нормативных решений:

* ordered;
* unordered;
* explicit order;
* ordered by default.

Открытые массивы внутри расширяемых объектов также имеют определённую семантику.

Canonical Change implementation больше не должна самостоятельно выбирать правила сравнения коллекций.

Блокер, выявленный в документе **Canonical Change Implementation Discovery**, считается устранённым.

```text
CANONICAL QA MODEL V0.1 COLLECTION SEMANTICS ESTABLISHED
```

```text
IMPLEMENTATION BLOCKER RESOLVED
```
