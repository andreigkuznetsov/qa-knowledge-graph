# Impact Evidence

> Детерминированный анализ влияния изменений в QA на основе проверенных данных, их происхождения и явно заданных границ применимости.

## Что такое Impact Evidence

**Impact Evidence** — платформа, которая получает подтверждённое изменение и определяет, какие выводы о его влиянии допустимы в пределах доступных инженерных данных. Она не ограничивается поиском связей: для каждого вывода учитывает происхождение, актуальность, согласованность и заявленную полноту данных.

Результат анализа — одна из трёх классификаций:

- `AFFECTED` — влияние подтверждено квалифицированными данными;
- `NOT_AFFECTED` — отсутствие влияния обосновано в явно заданных границах;
- `UNKNOWN` — данных недостаточно, чтобы подтвердить или исключить влияние.

Каждая классификация сопровождается основанием: цепочкой данных для `AFFECTED`, положительным обоснованием исключения для `NOT_AFFECTED` или причиной неопределённости для `UNKNOWN`. При одинаковом подтверждённом изменении, наборе данных и правилах анализа результат детерминирован и воспроизводим.

Контекст анализа включает не только сами отношения между артефактами. Для данных должны быть известны источник, применимый снимок, область действия и правила, по которым отношение участвует в распространении влияния. Если источник заявляет полноту, это утверждение относится только к конкретным категориям артефактов, типам отношений и моменту времени — оно не превращается в предположение о полноте всей инженерной модели.

Итоговый отчёт сохраняет этот контекст вместе с классификациями. Поэтому его можно проверить независимо: установить, какое изменение анализировалось, какие данные использовались, где действовал вывод и какое ограничение не позволило сделать более определённое заключение.

## Зачем нужен отдельный продукт

Программные системы связывают бизнес-правила, требования, сценарии, API, сервисы, данные и тесты. Когда один артефакт изменяется, инженеру нужно понять не только какие объекты с ним связаны, но и насколько надёжен вывод о влиянии на каждый из них.

Системы трассировки, анализаторы зависимостей и графы знаний хорошо решают собственные задачи: хранят отношения, находят пути и показывают связанный контекст. Однако найденный путь ещё требуется интерпретировать, а отсутствие пути может быть следствием неполной трассировки, устаревших сведений, конфликта источников или неразрешённой идентичности.

Без явной оценки этих условий результат выглядит определённее, чем позволяют исходные данные. Impact Evidence вводит между поиском связей и последующим инженерным решением отдельный результат: обоснованную классификацию с видимыми ограничениями.

Это особенно важно при работе с частично формализованной инженерной картиной. Неполная модель всё ещё может подтвердить отдельный путь влияния и принести практическую пользу. Проблема возникает не из-за неполноты как таковой, а когда неизвестная полнота скрыта и используется для исключающего вывода. Платформа позволяет работать с доступными сведениями, не приписывая им большей доказательной силы.

Тем самым решаются три самостоятельные задачи: вывод отделяется от простого обнаружения отношения, ограничения данных становятся частью результата, а downstream-потребители получают единый смысл классификаций вместо собственной неявной интерпретации каждого списка зависимостей.

## Логика анализа

Основной принцип платформы:

> **Инженерный вывод не может быть сильнее данных, на которых он основан.**

Из него следует различие между четырьмя уровнями:

1. **Обнаруженная связь** сообщает, что источник содержит отношение между артефактами.
2. **Инженерные данные** добавляют сведения об источнике, происхождении, актуальности, области действия и семантике отношения.
3. **Обоснованный вывод** фиксирует, что эти данные достаточны для одной из допустимых классификаций.
4. **Инженерное решение** определяет дальнейшее действие и остаётся за внешним потребителем.

Переход между уровнями не является автоматическим. Тип отношения должен иметь заранее определённый смысл для анализа; источник должен быть применим к рассматриваемому артефакту; происхождение данных должно позволять восстановить их путь до результата. Наличие записи в графе без этих условий остаётся обнаруженным фактом, но не обязательно становится основанием для классификации.

Для подтверждения влияния нужна применимая цепочка проверенных данных. Для обоснования отсутствия влияния требования строже: область анализа должна быть явно ограничена, а полнота релевантных данных — подтверждена. Если эти условия не выполнены, корректным результатом становится `UNKNOWN`.

Неопределённость может быть вызвана, например:

- отсутствующей трассировкой;
- устаревшим источником;
- противоречием между источниками;
- неразрешённой идентичностью артефакта;
- неизвестной полнотой данных;
- выходом вопроса за объявленные границы анализа.

Причина сохраняется в отчёте и показывает, какой именно предел данных достигнут. Она не является оценкой вероятности влияния и не ранжирует неопределённые объекты по важности.

Таким образом, платформа отвечает не на вопрос «что с чем связано?», а на вопрос «какие выводы допустимы на основании этих данных и в этих границах?».

## Пример

Предположим, Canonical Change подтвердил изменение бизнес-правила проверки возраста клиента при регистрации. Для анализа доступны актуальные данные из известного источника трассировки, а смысл поддерживаемых отношений задан заранее.

Источник идентифицирован, его снимок относится к анализируемой версии модели, а происхождение каждой используемой связи можно восстановить. При этом заявление о полноте относится только к отдельно обозначенной части модели. Эти условия важны: именно они определяют, где допустим положительный или исключающий вывод.

В модели рассматриваются три сценария:

- **Сценарий A** связан с изменённым правилом квалифицированной цепочкой трассировки;
- для **Сценария B** связь не найдена, но полнота соответствующей части трассировки не подтверждена;
- **Сценарий C** входит в область, для которой источник явно подтверждает полный набор релевантных отношений, и данные обосновывают его исключение из влияния.

Результат:

| Объект | Классификация | Основание |
|---|---|---|
| Бизнес-правило | `AFFECTED` | Подтверждённое изменение относится непосредственно к правилу |
| Сценарий A | `AFFECTED` | Существует применимая цепочка проверенных данных от изменённого правила |
| Сценарий B | `UNKNOWN` | Связь не найдена, но полнота трассировки в этой области неизвестна |
| Сценарий C | `NOT_AFFECTED` | Исключение обосновано в пределах явно полной и актуальной области данных |

Для Сценария B платформа не угадывает наиболее вероятный ответ. Для Сценария C она также не делает глобального заявления: `NOT_AFFECTED` действует только в указанной области данных и для рассматриваемого изменения.

Если позднее появится новый снимок трассировки, анализ будет относиться уже к другому контексту. Старый отчёт останется воспроизводимым свидетельством того, какой вывод был обоснован доступными на тот момент данными.

## Отличие от традиционного анализа влияния

| Вопрос | Традиционный анализ | Impact Evidence |
|---|---|---|
| Что является исходным результатом? | Найденные зависимости или связанные объекты | Квалифицированные выводы о влиянии |
| Как учитывается происхождение данных? | Зависит от конкретного инструмента | Сохраняется для каждого вывода |
| Что означает отсутствие пути? | Нередко интерпретируется вручную | Оценивается вместе с полнотой и границами данных |
| Когда допустим вывод об отсутствии влияния? | Часто не формализовано | Только при наличии положительного основания для исключения |
| Видна ли неопределённость? | Не всегда | Представлена классификацией и конкретной причиной |
| Можно ли воспроизвести вывод? | Зависит от сохранённого контекста | Контекст анализа является частью результата |

Impact Evidence не заменяет существующие инструменты и не объявляет найденную связь причинностью. Он использует предоставленные ими инженерные данные, проверяет, какие выводы они позволяют сделать, и сохраняет основания этих выводов.

Это отличие не делает традиционный анализ менее ценным. Статический анализ точнее отвечает на вопросы о кодовых зависимостях, трассировка хранит явные отношения требований и тестов, а граф знаний связывает разнородный инженерный контекст. Impact Evidence не повторяет их работу: его продуктовый результат начинается там, где найденные сведения требуется квалифицировать для ограниченного вывода о влиянии.

## Место в инженерной экосистеме

Платформа располагается между источниками инженерных данных и системами, которые принимают решения:

```text
Canonical Change: подтверждённое изменение
                    ↓
Трассировка, анализ зависимостей, графы знаний
                    ↓
Impact Evidence: AFFECTED / NOT_AFFECTED / UNKNOWN
                    ↓
Внешние потребители результата
```

**Canonical Change** отвечает за корректность изменения и формирует неизменяемый `VerifiedChangeSet`. Impact Evidence не повторяет эту проверку.

**Системы трассировки, анализаторы зависимостей и графы знаний** предоставляют сведения об артефактах и отношениях. Impact Evidence не заменяет их, не становится новым хранилищем и не восстанавливает отсутствующие связи.

**Downstream-системы и инженеры** используют отчёт вместе с другими данными и собственными правилами. К ним относятся механизмы принятия решений, планирование регрессии, оценка готовности релиза, аудит и контроль соответствия.

Каждый такой потребитель получает одну и ту же доказательную основу, но использует её по-разному. Планирование может сопоставить её с покрытием и стоимостью выполнения тестов; оценка готовности — с результатами сборки и контролями выпуска; аудит — с требованиями к прослеживаемости. Эти дополнительные сведения и правила не входят обратно в классификацию Impact Evidence.

Граница ответственности проходит между выводом и действием. Impact Evidence не выбирает тесты, не определяет объём регрессии, не рассчитывает риск, не оценивает готовность релиза, не формирует рекомендации и не принимает инженерные решения.

Этот README знакомит с назначением и границами Impact Evidence. Подробные аспекты платформы распределены по отдельным документам:

Статус документов описывает текущее состояние проектирования, а не обещание конкретных функций или сроков выпуска. До стабилизации публичного контракта README следует рассматривать как описание назначения и границ продукта.

## Принципы проектирования

- детерминированность результатов;
- воспроизводимость анализа;
- сохранение происхождения данных;
- явные границы применимости;
- объяснимость каждой классификации;
- независимость от конкретного способа хранения инженерных данных;
- совместимость с существующей инженерной экосистемой.

## Участие в проекте

На текущем этапе особенно полезны:

- экспертная обратная связь по инженерной модели;
